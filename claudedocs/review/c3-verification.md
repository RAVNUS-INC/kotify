# C3 — 29002 재시도 중복발송 외부 검증 (반증)

> 목적: P2-C3(🔴) "29002 재시도가 채널 바꿔 재발송 → 부분수락분 중복"을 LG U+ msghub
> 공식 스펙으로 끝까지 확인해 확정/반증.
> 원칙: C4 교훈 — 코드만 보면 방향이 틀린다. 외부 스펙 선확인.
> 일자: 2026-05-30 · 출처: https://doc.msghub.uplus.co.kr/guide/d/결과-코드 , /통합메시지

---

## 판정: ❌ 반증 (false positive) — 🔴 철회

리뷰어의 재현 시나리오("일부 수신자는 큐잉되고 한도 초과분만 거절 후 전체 코드 29002")는
**가능성** 제기였고, 공식 스펙이 이를 반증한다. 29002 는 요청 전체 거부이므로 접수된
수신자가 0 → `-fb` 직접 재발송이 중복 발송할 대상이 없다.

---

## 근거 (외부 스펙 + 코드 교차)

1. **CPS = Calls Per Second** — 수신자가 아니라 **API 호출(요청) 단위** 레이트리밋.
   단일 요청이 초당 호출 한도를 넘으면 그 요청이 거부되는 것이지, 요청 내 수신자가
   부분 처리되는 개념이 아니다.

2. **29002 는 HTTP 400 + 공통(common) 최상위 code** (공식 결과코드 표).
   - 발송 API 응답 구조: `{ "code": <요청결과>, "message", "data": [ {cliKey, msgKey, phone, "code": <수신자별>, ...} ] }`.
   - **부분수락은 HTTP 200 + 최상위 code=10000** 일 때 `data[].code` 로만 표현된다(요청은 통과, 개별 수신자 실패 — 이것이 H1 이 다룬 경로).
   - 29002 는 최상위 code 가 비성공(HTTP 400)이라 요청이 ingress 에서 통째로 거부됨 — `data` 단계(수신자 처리)에 도달하지 않는다.

3. **앱 코드도 최상위 code 로만 분기**: `client.py:_raise_for_response` 는
   `code = body.get("code")`(요청 결과)만 보고 `code == "29002"` 일 때 `MsghubRateLimited`
   raise(client.py:58,69). 즉 `MsghubRateLimited` 는 **요청 전체 29002** 에서만 발생.

4. **결론**: 첫 RCS 시도가 29002 면 접수된 수신자 0 → 30초 후 `-fb` 직접 SMS 재시도가
   동일 수신자에게 2번째 메시지를 보낼 수 없다. 중복 발송·이중 과금 위험 없음.

---

## 조치

- 코드 무변경(현 retry 로직 정상). `compose.py` MsghubRateLimited 핸들러에 확정 의미론
  주석 추가 — 동일 우려 재발 방지.
- 미검증이던 레이트리밋 재시도 경로를 회귀 테스트로 고정: `tests/test_dispatch_ratelimit.py`
  (29002 → SMS 재시도 → 수신자당 메시지 정확히 1건, `-fb` 키, 실패 0).

## 잔여 관찰 (🔴 아님, 별도 백로그)

- **정책 일관성(🟡)**: 29002 만 즉시 채널변경(`-fb` 직접 SMS) 재발송하고, 다른 retryable
  (5xx·49xxx·타임아웃)은 fail-and-record 다(phase2:64). 레이트리밋은 동일 채널(RCS)
  멱등 재시도가 의미상 더 정확하나, **중복·유실 버그가 아닌 품질 개선**이라 본 검증 범위 밖.
  채널 변경의 부수효과(RCS→SMS 강등, 단가/사용자경험 차이)를 감안해 별도 판단 권장.
